Case 1

식품 공정 특화 예외 처리 구조 설계

문제 상황
  • 식품업 특성상 원자재 상태 변동이 잦아 반품 후 재입고, 혼합(Mixing), 재작업 등의 복잡한 예외 발생
  • 기존 시스템으로는 이러한 흐름을 반영하기 어려워 데이터 왜곡 발생
해결 방법
  • 반품 -> 재입고 -> 재투입 흐름을 하나의 상태 전이 구조로 설계
  • Mixing 공정을 반영하여 여러 Lot이 하나의 공정으로 결합되는 구조 구현
  • 각 공정에서 발생하는 상태 변화를 데이터 단위로 관리
성과
  • 복잡한 공정 흐름에서도 데이터 일관성 유지
  • Lot 단위 추적 정확도 향상
  • 예외 상황 대응으로 실제 운영 환경에서 추가 수정 없이 안정적 사용
  • 사전 예외사항을 반영해 초기 적용 단계에서도 추가 수정 및 오류 발생을 최소화하며 운영 안정성 확보
Case 2

실시간 처리 구조 개선 (Polling -> SSE)

문제 상황
  • 기존 폴링 방식으로 데이터를 조회하며 상태 반영 지연 및 불필요한 요청 발생
  • 공정 상태가 즉시 반영되지 않아 현장 대응 속도 저하
해결 방법
  • 폴링 구조를 제거하고 SSE(Server-Sent Events) 기반 실시간 이벤트 처리 구조로 전환
  • RFID 이벤트 발생 시 서버에서 클라이언트로 즉시 데이터 전달하도록 설계
성과
  • 상태 반영 시간 최대 수 초 -> 1초 이내로 단축
  • 불필요 요청 약 60% 감소 및 서버 부하 완화
  • 실시간 공정 모니터링 가능 -> 현장 대응 속도 향상
Case 3

공정 단절 및 데이터 불일치 문제 해결

문제 상황
  • 공정별 시스템이 분리되어 자재-생산-출하 간 데이터 단절 발생
  • 동일 데이터가 여러 번 입력되며 불일치 및 수동 보정 작업 발생
해결 방법
  • 자재-공정-완제품-출하까지 이어지는 End-to-End 데이터 흐름 구조 설계
  • 공정 단위 데이터를 기준으로 흐름을 재정의하고, 상태 변경이 자동으로 다음 공정에 반영되도록 설계
성과
  • 데이터 불일치로 인한 수동 보정 작업 약 80% 이상 감소
  • 공정 간 데이터 단절 해소 -> 생산 이력 추적 가능
  • 초기 운영 안정화 기간 단축 및 추가 수정 요청 최소화
